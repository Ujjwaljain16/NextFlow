Visual Workflow Builder - System Architecture
1. Core Architecture Principles
Inspired by production-grade Next.js systems like Dub.

Thin Routing / Thick Services: Route Handlers (app/api/) only handle HTTP, auth (Clerk), and request parsing (Zod). All business logic lives in lib/services/.
Domain-Driven Grouping: Features are namespace-isolated (e.g., src/nodes/, src/trigger/).
Adapter Pattern for Third-Parties: External dependencies (Trigger.dev, Uploadcare, Cloudinary) are wrapped behind strict internal interfaces.
Zod at the Boundaries: Strict runtime validation for API routes, webhooks, and Node execution inputs.
2. High-Level Architecture
UI Layer (src/components, src/app): React Flow canvas and Tailwind/ShadCN UI components. No direct data mutations; relies on custom hooks and Zustand.
Workflow Engine (src/lib/services/workflow.service.ts): Handles CRUD and converts visual graph coordinates into execution DAGs.
Execution Orchestrator (src/lib/services/orchestrator.service.ts): Centralized state machine that evaluates the DAG, handles fan-out (parallel dispatch), and fan-in (dependency waiting).
Node Runners (Trigger.dev): Isolated background workers that fetch configs, execute safely, and report outputs back to the Orchestrator.
Input Resolver (src/lib/services/resolver.service.ts): Reshapes outputs based on edge mappings before injecting them into target nodes.
3. Data Models (PostgreSQL & Prisma)
Workflow Schema (The Blueprint)

id, userId, name, status
nodes: Array of { id, type, data, position }
edges: Array of { source, target, sourceHandle, targetHandle }
WorkflowRun Schema (The Instance)

id, workflowId, userId, status (PENDING, RUNNING, SUCCESS, FAILED, CANCELED)
Execution Graph (Precomputed JSON):
adjacencyList: { "NodeA": ["NodeB"] }
inDegreeMap: { "NodeB": 1 }
startedAt, completedAt
NodeRun Schema (The Tracker & Dependency State)

id, workflowRunId, nodeId
status (PENDING, RUNNING, SUCCESS, FAILED)
inputs: JSON
outputs: JSON (Pointers/URIs for large files via Transloadit, never raw binary)
error: String
dependenciesMet: Integer (Replaces MongoDB $addToSet, protected by row-level locking)
4. Execution Orchestrator & Trigger Mechanism
Model: Centralized Orchestrator Loop.
Trigger: NodeExecutionTask finishes and natively emits a node.completed custom event.
Batching: The Orchestrator subscribes to this event as a batch trigger (e.g., batchSize: 50, batchWindowMs: 500) to process multiple parallel completions simultaneously, preventing DB write contention.
5. Concurrency & Idempotency Safeguards
Unique Constraints: NodeRun table has a compound unique constraint on @@unique([workflowRunId, nodeId]) to prevent duplicate executions from race conditions.
Safe Dependency Counting: Since PostgreSQL lacks MongoDB's atomic $addToSet on nested JSON, we track readiness via dependenciesMet explicitly on the target NodeRun record using Prisma Database-level transactions and Row-Level Locks (SELECT ... FOR UPDATE).
Strict State Transitions: DB queries enforce allowed states (WHERE status = 'PENDING'). Terminal states (SUCCESS, FAILED) can never be overwritten by late-arriving events.
Partial Continuation: A failed node causes downstream branches to die gracefully (Waiting indefinitely), while parallel independent branches continue executing.
6. Input Resolution & Data Flow
Large Data Protocol: Blobs are uploaded securely via Transloadit; nodes only pass JSON pointers to URIs through Trigger payloads.
Type Safety: Nodes define exact Zod inputSchema schemas. The ResolverService normalizes, coerces, and validates mapped data before passing it to Trigger.dev.
